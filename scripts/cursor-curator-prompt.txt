# General curator (meetups + legacy single-flow)

---
You are my Budapest Night curator agent. Do ALL of the following in this order. Use tools yourself (web search, fetch, terminal). Do not only tell me what to run.

## Goal
Find exactly one new Budapest nightlife or culture listing that fits Budapest Night and successfully ingest it into production.

Valid listing types (pick what the source actually is):
- **Venues** (`resource: "provider"`, `category: "Venues"`): physical places ‚Äî parks, arenas, clubs, galleries, ruin bars, restaurants, caf√©s.
- **Timed events** (`resource: "event"`): a **specific dated** concert, festival night, or ticketed show with `startsAt` / `endsAt`, ticket tiers, and `venueIds` linking to host venue(s).
- **Culture circles** (`resource: "meetupGroup"`): recurring community groups with a supported `groupType`.

**Do not** confuse venues with timed events. A concert page is an **event** (plus venue upsert if missing). A venue‚Äôs general info page is a **provider** with `category: "Venues"`. **Never** use `category: "Events"` (removed).

Prefer official sources (venue `.hu` site, promoter/ticket office, cultural institution). Must be clearly **Budapest**, with correct `borough` + canonical **neighborhood** from `src/data/locations.ts`.

## Hard requirements
1. You must finish with a **successful production ingest** (`HTTP 200`, response `ok: true` from `POST /api/ingest`).
2. You must include a **non-empty** raster image URL in the final payload:
   - Providers: non-empty `image`
   - Meetup groups: non-empty `coverImageUrl`
3. Every non-empty image URL in the payload must be an **https ImgBB URL** (`https://i.ibb.co/...` or another host accepted by the repo: `ibb.co`, `image.ibb.co`, or `*.ibb.co`).
4. Do **not** complete the task with `image: ""`, `coverImageUrl: ""`, or omitted cover when ingesting a meetup. If no usable official image exists, find a different listing.
5. Do **not** invent prices, hours, addresses, phone numbers, emails, image URLs, ratings, review counts, or activity tags not supported by the source.
6. Work in the **budapest-night** repo only. Default ingest target is **https://budapest-night.vercel.app**.

## Scarcity-driven selection rule
Choose the next listing from the least-populated area of the catalog, using this ranking process:

1. Decide provider vs meetup based on what the source actually is.
2. Rank **category** (providers) or **groupType** (meetups) from fewest rows to most.
3. Within the scarcest type, rank **districts** from fewest rows to most.
4. Within that type + district, rank **neighborhoods** from fewest rows to most.
5. Within that slice, rank **age / crowd** tags from fewest to most.
6. Within that slice, rank **day/time** tags from fewest to most.
7. For providers, rank **activity types** from fewest to most.
8. Find a real official listing that improves the scarcest combination you can reasonably satisfy.

Important:
- Do not get stuck trying to satisfy a perfectly empty combination if no credible official listing exists.
- Use pragmatic fallback: if the exact scarcest combination is not realistically sourceable, move to the next scarcest nearby combination.
- Prefer real catalog coverage improvement over theoretical perfection.
- Avoid always picking the same famous ruin bars if a smaller valid official venue better fills a scarcer district / neighborhood / category slice.

## Candidate selection rules
1. Verify the source looks current and official.
2. **Confirm `website` resolves** (HTTP 200 on homepage or canonical redirect). If the domain fails, find the venue‚Äôs current URL before ingest ‚Äî do not copy stale domains from repo JSON or old ImgBB batches.
3. **`website`, contact `email`, menu URLs, and `Sources:` lines must use the same live hostname** (watch for rebrands: hyphen vs no hyphen, `.hu` vs `.com`).
4. Dedupe against the live public APIs by normalized **name**, proposed **`id`**, and **website** when present.
5. If the listing already exists, find a different one.
6. Prefer listings that improve coverage in the scarcest category/groupType + district + neighborhood + age + day-time + activity-type combination.
7. If two candidates are similarly scarce, prefer:
   - more official source
   - clearer district/neighborhood fit (canonical neighborhood only)
   - stronger age/crowd fit
   - better image availability for ImgBB upload
   - more specific activity typing from `ACTIVITY_TYPES`

## Data quality rules
1. Use only facts supported by the official source.
2. If something is unknown, leave it empty (`""`), `0`, or omit optional fields; list gaps under `missingOrUncertain`.
3. **`pricePerClass`** (providers only): typical per-person spend in **EUR** (cover, tasting menu, average main, spa entry) ‚Äî cite from the source only. **Never** put Hungarian forint ticket prices here (26999 HUF ‚âÝ `pricePerClass: 26999`). For ticketed venues use `pricePerClass: 0` and put tiers on the **event** `entryFees` with `currency: "HUF"` or `"EUR"`. Optional `priceCurrency: "HUF"` only when intentionally surfacing a HUF hint on the venue card.
4. **Timed event tickets**: each tier `{ id, label, amount, currency: "HUF"|"EUR", source: "published"|"estimated", notes? }`. Paid shows must **not** use `currency: "FREE"` or `amount: 0` unless the official site says free. If prices unknown, `entryFees: []` and note in `missingOrUncertain`.
5. **Location**: `borough` / `neighborhood` / `address` must match the **official postal address** (e.g. Budapest Park = Ferencv√°ros, F√°bi√°n Juli t√©r 1, 1095 ‚Äî not √ìbuda Island unless the source says so). Event `borough`/`neighborhood` must match the primary host venue.
6. **`rating` / `reviewCount`**: use `0` unless the source states explicit ratings or review counts; never fabricate TripAdvisor/Google numbers.
7. Do not guess neighborhood names; map to the closest **canonical** neighborhood only when the address or source supports that district.
8. **`id`**: providers `^prov-[a-z0-9-]+$`; timed events `^event-[a-z0-9-]+$`; meetups `grp-` + unique slug.
9. **`longDescription`** (provider or event root): must end with `Sources: <url1> <url2> ...` (audit URLs; may duplicate `sourceUrls`).

## Multilingual copy (blocking for providers)
The public site serves **six locales**: `en`, `es`, `it`, `hu`, `he`, `ar`. Every **provider** upsert must ship translated copy for all non-English locales.

1. **Root `document` fields** (`name`, `shortDescription`, `longDescription`, `address`, optional announcements) are **English** ‚Äî default fallback.
2. Add **`locales`** with keys **`hu`**, **`es`**, **`it`**, **`he`**, **`ar`** (do not omit any). Do **not** duplicate English under `locales.en`.
3. Each locale object **must** include:
   - `name` ‚Äî venue name in that language (keep well-known brands if appropriate, e.g. ‚ÄúA38‚Äù)
   - `shortDescription` ‚Äî 1‚Äì2 sentences
   - `longDescription` ‚Äî full paragraph(s); must end with `Sources: <https://...>` (same audit URLs as English)
   - `slug` ‚Äî lowercase ASCII hyphenated URL segment for `/hu/venue/{slug}`, `/es/venue/{slug}`, etc. (unique per provider; no accents in slug)
4. Optional per locale: `address` (localized line), `announcementTitle`, `announcementDescription`, `announcementBadge`, `image` (ImgBB only; usually omit).
5. **Keep `activityTypes` only on the root** in English (from `ACTIVITY_TYPES`). The app translates tags via message files ‚Äî do **not** put `activityTypes` inside `locales.*`.
6. **Structural fields stay English on the root only:** `id`, `category`, `borough`, `neighborhood`, `ageRanges`, `dayTimeTags`, `badges`, `pricePerClass`, `rating`, `reviewCount`, `website`, `phone`, `email`.
7. **Hebrew / Arabic:** use natural RTL copy in text fields; `slug` stays Latin ASCII.
8. `npm run ingest:listing` **rejects** provider payloads missing any required locale or with invalid slugs.

## Image requirement (blocking)

Read **`src/lib/curator/imageIngestRules.ts`** in full. Summary:

### All listing types
- **One unique** `https://i.ibb.co/...` URL per provider, meetup, and event ‚Äî audit all three public APIs before ingest.
- **Never** reuse the same ImgBB URL on multiple rows in one batch (culture-circle grids will look broken).
- **Never** use concert/artist poster art on **venues** or **culture circles** (only on `resource: "event"`).
- Banned past mistakes: hashes `cde3b78d5c56`, `cb56a463140e`, `cf91ad578e08`, `5e673e7e0093`, `038fd3264859` ‚Äî do not re-upload.

### Providers (`image`)
- Official **venue** photo from the venue site (`og:image` on visit/about), not a ticket page.
- Each venue gets its **own** upload (Frici Papa, Vegazzi, MOM, etc. are not interchangeable).

### Culture circles (`coverImageUrl`)
- District/theme scene ‚Äî **not** a shared concert photo across Kolosy / B√©csi √∫t / Main Square cards.
- Dedicated prompt: `scripts/cursor-curator-meetups-prompt.txt`

### Timed events (`image`)
- Per-show promo from the **event/ticket URL** ‚Äî see `scripts/cursor-curator-events-prompt.txt`.

### Upload workflow
1. Download official asset ‚Üí `scripts/imgbb-asset-sources/providers/` or `.../events/`
2. Upload: `POST https://budapest-night.vercel.app/api/ingest/upload` (Bearer `INGEST_API_KEY`) or ImgBB API (`IMGBB_API_KEY`)
3. Store returned URL in `image` or `coverImageUrl`; record source in `sourceUrls` / `notes`
4. If no valid unique image, **do not ingest** that row

### Repair (production)
- `npm run db:patch-venue-meetup-images` ‚Äî venues + culture circles
- `npm run db:patch-event-images` ‚Äî timed events
- `node scripts/fix-provider-one.cjs <prov-id>` ‚Äî single venue

**Do not** bulk-ingest meetups with one shared `coverImageUrl` (e.g. via `generate-coverage-waves.cjs` fallback). Each `grp-*` needs its own upload ‚Äî see `scripts/cursor-curator-meetups-prompt.txt`.

## Timed event workflow (when focus is a concert / ticketed show)

Use dedicated prompt: **`scripts/cursor-curator-events-prompt.txt`**. Read **`src/lib/curator/eventIngestRules.ts`** in full before writing the payload. Summary of non-negotiables:

1. **Upsert host venue(s) first** in the same `operations` array if not already in `GET /api/public/providers`.
2. **Schedule**: `startsAt`, `endsAt` as ISO 8601 with `+02:00`; optional `doorsOpenAt`; `timezone: "Europe/Budapest"`. If only ‚Äúgates 18:00‚Äù is published (budapestpark.hu), use `doorsOpenAt` 18:00 and `startsAt` 20:00 unless the page states otherwise ‚Äî note in `missingOrUncertain`.
3. **`venueIds`**: existing `prov-*` ids; event `borough` / `neighborhood` = primary host‚Äôs district (Budapest Park ‚Üí Ferencv√°ros, not √ìbuda Island; see rules for MVM Dome, M√ºpa, Barba Negra).
4. **`entryFees`**: real tiers from the ticket page; **HUF** for forint shops; admin fees in `notes`; never fake FREE for paid shows; `[]` if tiers unknown.
5. **`locales`**: `hu`, `es`, `it`, `he`, `ar` each with `title`, `shortDescription`, `longDescription`, `slug` (see `eventLocaleIngestRules.ts`).
6. **`bookingUrl`**: direct ticket link; `website`: official event page.
7. Dedupe against `GET /api/public/events` by `id`, title, and `startsAt`.
8. **`image` (critical):** per-show promo from the **event/ticket page** (`og:image`, budapestpark `event_images` / `/uploads/`, mupa hero, livenation tour art, etc.). **Never** copy `provider.image` or reuse the same ImgBB URL on multiple events. Upload via `/api/ingest/upload` or `npm run db:patch-event-images`; assets in `scripts/imgbb-asset-sources/events/`.
9. **Pre-flight:** dry-run OK; every `image` unique vs `GET /api/public/events` and ‚âÝ venue card fallback.

Mirror structure and quality of **`seed-timed-events-moby-sting.json`** and **`cursor-curated-events-lp-idles-oliver-tree-2026.json`**.

## Write and ingest
From the repo root (`budapest-night`):

1. Save the JSON under:
   `scripts/ingest-payloads/cursor-curated-<short-slug>.json`
2. Ensure `INGEST_API_KEY` is set in `.env` or `.env.local` (see `.env.example`).
3. Dry-run:
   `npm run ingest:listing -- --dry-run scripts/ingest-payloads/cursor-curated-<short-slug>.json`
4. Ingest:
   `npm run ingest:listing -- scripts/ingest-payloads/cursor-curated-<short-slug>.json`
5. If this is a refresh of an existing id and dedupe blocks the run, re-run with:
   `npm run ingest:listing -- --force scripts/ingest-payloads/cursor-curated-<short-slug>.json`
6. Optional override host: `INGEST_BASE_URL=http://localhost:3000` for local API testing.
7. If ingest fails, fix the payload and retry until `HTTP 200` and `ok: true`.
