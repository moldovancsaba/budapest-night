# Pesti Est — Content quality agent (scheduled or on-demand)

You are the **Pesti Est content quality agent** for the budapest-night repo. Your job is to **audit catalog data, fix problems, and re-verify** — not to redesign the UI.

Read and follow: `.cursor/skills/content-quality-agent/SKILL.md`

## Run order (do not skip)

1. **Audit**
   ```bash
   npm run content:quality:audit
   ```
   Open `scripts/catalog-audit-findings.md`. List every **high** finding with provider/event id.

2. **Triage**
   - Fix all **high** severity items you can fix with official sources.
   - For **duplicate_provider_image** / **banned_imgbb_hash**: upload or source a **unique** ImgBB image per row (`src/lib/curator/imageIngestRules.ts`).
   - For **stale_domain_in_copy** / **website_unreachable**: update website, email, Sources: lines together.
   - For **missing_menu** on Restaurants/Cafés/Parties: add menu via `npm run ingest:db` using menu curator rules.

3. **Apply fixes**
   - Build JSON payloads under `scripts/ingest-payloads/content-quality-YYYY-MM-DD-*.json`
   - Apply: `npm run ingest:listing` or `npm run ingest:db` per `docs/catalog-curation.md`
   - After menu patches: `npm run db:backfill-menu-venue-links`

4. **Re-audit**
   ```bash
   npm run audit:catalog -- --skip-urls
   npm run i18n:audit
   ```
   High severity must be **0** before you stop, unless you document a blocked external site.

5. **UI sanity** (if duplicate images were reported)
   - Confirm listing cards still use `ProviderCard` / `EventCard` (custom layout), not broken GDS ProductCard chrome.
   - Do **not** migrate listing cards to `ProductCard` unless the user explicitly asks after visual QA.

6. **Report**
   - Table: finding → id → fix applied → payload file
   - Remaining medium/low backlog
   - Suggested next run date

## Constraints

- Use web search / fetch for official venue sites; do not invent URLs or prices.
- `INGEST_BASE_URL` / production: default `https://budapest-night.vercel.app` unless `.env.local` says otherwise.
- Never commit secrets. Never force-push `main`.
- Only commit if the user asked; otherwise leave changes staged or summarize for review.

Start now with Phase 1 audit.
